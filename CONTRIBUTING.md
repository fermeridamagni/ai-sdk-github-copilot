## Code of Conduct

All contributors are expected to be respectful and constructive. Harassment,
personal attacks, or discriminatory language of any kind will not be tolerated.
Please follow the [GitHub Community Guidelines](https://docs.github.com/en/site-policy/github-terms/github-community-guidelines).

## Prerequisites

Before contributing, make sure you have the following installed and configured:

| Requirement | Version / Notes |
|---|---|
| [Bun](https://bun.sh) | 1.0 or later — used as the package manager and runtime |
| [Node.js](https://nodejs.org) | 18 or later (required by some tooling) |
| [GitHub Copilot CLI](https://github.com/github/copilot-sdk) | Install globally with `npm i -g @github/copilot` and authenticate with `gh auth login` |
| A GitHub account with an active Copilot subscription | Required to run end-to-end tests against real Copilot sessions |
| [Git](https://git-scm.com) | For version control |

> **Why Bun?** The project rules mandate Bun as the sole package manager and
> runtime. Do not use `npm`, `pnpm`, or `yarn` — they may produce lock-file
> conflicts that break CI.

## Setting Up the Development Environment

```sh
# 1. Fork the repository and clone your fork
git clone https://github.com/<your-username>/ai-sdk-github-copilot.git
cd ai-sdk-github-copilot

# 2. Install dependencies (Bun resolves bun.lock)
bun install

# 3. Verify the Copilot CLI is accessible and authenticated
gh copilot --version   # prints the CLI version
gh auth status         # shows your active GitHub session

# 4. Confirm the build passes
bun run build

# 5. Confirm lint passes
bun x ultracite check
```

If `bun x ultracite check` reports issues, run `bun x ultracite fix` to
auto-correct them before proceeding.

## Development Workflow

1. **Create a branch** off `main` using the naming conventions below.
2. **Make your changes** — keep commits small and focused.
3. **Run the linter & formatter** before every commit:
   ```sh
   bun x ultracite fix   # auto-fixes most style issues
   bun x ultracite check # confirms no remaining issues
   ```
4. **Run the tests** to make sure nothing is broken:
   ```sh
   bun run test
   ```
5. **Build the package** to verify TypeScript compilation succeeds:
   ```sh
   bun run build
   ```
6. **Open a pull request** against `main` with a clear description of what
   changed and why.

## Code Standards

This project uses [Ultracite](https://ultracite.dev) — a zero-config preset
built on [Biome](https://biomejs.dev) — for automated formatting and linting.

### Quick commands

| Command | What it does |
|---|---|
| `bun x ultracite fix` | Auto-formats and fixes most lint issues |
| `bun x ultracite check` | Reports issues without modifying files |
| `bun x ultracite doctor` | Diagnoses Ultracite/Biome setup |

### Style rules to keep in mind

- **TypeScript only** — no plain JavaScript files.
- **Explicit types** on public function parameters and return values.
- **`unknown` over `any`** when a type is genuinely unknown.
- **`const` by default**, `let` only when reassignment is needed; never `var`.
- **`async/await`** instead of `.then()`/`.catch()` chains.
- **`for...of`** instead of `.forEach()` or indexed `for` loops.
- **Early returns** to reduce nesting; prefer guard clauses.
- **No `console.log`/`debugger`** in production code.
- **Descriptive names** — avoid magic numbers and cryptic abbreviations.
- Throw `Error` objects with clear messages, not bare strings.

For React/JSX or framework-specific guidance, see [`AGENTS.md`](./AGENTS.md).

## Running Tests

```sh
# Run all tests once
bun run test
```

Tests live in the `tests/` directory and are powered by
[Vitest](https://vitest.dev).

### Writing tests

- Place test files alongside the unit they test or inside `tests/`.
- Use `it()` or `test()` blocks and `expect()` assertions — no done callbacks.
- Do **not** commit `.only` or `.skip` modifiers.
- For logic that requires a live Copilot session, gate the test with an
  environment check so CI can skip it when credentials are unavailable:

  ```ai-sdk-github-copilot/tests/example.test.ts#L1-12
  import { describe, it, expect } from "vitest";

  const hasCopilot = process.env.GITHUB_TOKEN !== undefined;

  describe.skipIf(!hasCopilot)("live Copilot tests", () => {
    it("generates a response", async () => {
      // ...
    });
  });
  ```

## Submitting a Pull Request

1. **Fork** the repository and create your branch from `main`.
2. Fill out the PR description template (what changed, why, how to test).
3. Ensure all checks pass:
   - `bun x ultracite check` reports no errors.
   - `bun run build` succeeds without TypeScript errors.
   - `bun run test` passes.
4. Keep the diff focused — one logical change per PR is strongly preferred.
5. Reference any related issues with `Closes #<issue>` or `Relates to #<issue>`.
6. A maintainer will review your PR and may request changes before merging.

### Branch naming

| Type | Pattern | Example |
|---|---|---|
| New feature | `feat/<short-description>` | `feat/tool-calling-bridge` |
| Bug fix | `fix/<short-description>` | `fix/stream-abort-cleanup` |
| Documentation | `docs/<short-description>` | `docs/contributing-guide` |
| Refactor | `refactor/<short-description>` | `refactor/session-lifecycle` |
| Tests | `test/<short-description>` | `test/generate-path-unit` |
| Chore / tooling | `chore/<short-description>` | `chore/update-biome` |

## Commit Message Conventions

This project follows [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/):

```
<type>(<optional scope>): <short summary>

[optional body]

[optional footer(s)]
```

### Types

| Type | When to use |
|---|---|
| `feat` | A new feature visible to users |
| `fix` | A bug fix |
| `docs` | Documentation only changes |
| `refactor` | Code restructuring with no behaviour change |
| `test` | Adding or updating tests |
| `chore` | Tooling, dependencies, CI |
| `perf` | Performance improvements |
| `build` | Changes to the build system or external dependencies |

### Examples

```
feat(stream): forward reasoning deltas as reasoning-delta stream parts
fix(generate): correctly handle abort signal during session wait
docs: add architecture diagram to README
chore: update @github/copilot-sdk to 0.4.0
```

## Reporting Bugs & Requesting Features

Please use [GitHub Issues](https://github.com/fermeridamagni/ai-sdk-github-copilot/issues).

### Bug reports should include

- A minimal reproduction (ideally a single TypeScript file).
- The exact error message and stack trace.
- Your Bun version (`bun --version`), Node version (`node --version`), and
  Copilot CLI version (`gh copilot --version`).
- The model ID you were using.

### Feature requests should include

- A description of the use case or problem you are trying to solve.
- Any relevant upstream links (e.g., new `@github/copilot-sdk` APIs, AI SDK
  stream part types).

## Architecture Notes

The core challenge this package solves is bridging two incompatible models:

| AI SDK (stateless) | Copilot SDK (session-based) |
|---|---|
| One `doGenerate` / `doStream` call → one result | Open a `CopilotSession`, send messages, collect events, close session |
| Multi-turn history passed as a `LanguageModelV3Prompt` array | Single prompt string per session message |
| `abort` signal cancels the request | Must explicitly stop / close the session |

### How the bridge works

1. **`createCopilot(settings)`** captures CLI/auth configuration and lazily
   creates a `CopilotClient` that is reused across requests.
2. **`doGenerate`** opens a `CopilotSession`, calls `session.sendAndWait`, maps
   the response to `LanguageModelV3GenerateResult`, and closes the session.
3. **`doStream`** opens a `CopilotSession`, subscribes to
   `assistant.message_delta` / `assistant.reasoning_delta` events, maps each
   delta to the corresponding AI SDK stream part, and closes the session on
   `session.idle` or `session.error`.
4. **`convertToCopilotPrompt`** serialises the full `LanguageModelV3Prompt`
   (system + user + assistant turns) into a single string forwarded as the
   user message, with the system content surfaced via
   `session.systemMessage = { mode: 'append', content: '...' }`.

### Key type dependencies

| Package | What we use |
|---|---|
| `@ai-sdk/provider` | `LanguageModelV3`, `LanguageModelV3CallOptions`, streaming part types |
| `@ai-sdk/provider-utils` | `NoSuchModelError`, utility helpers |
| `@github/copilot-sdk` | `CopilotClient`, `CopilotSession`, `CopilotClientOptions` |

## Roadmap / Good First Issues

These are known gaps that are good candidates for contributions:

| Area | Description | Complexity |
|---|---|---|
| **Tool calling** | Bridge AI SDK `tools` / `toolChoice` to Copilot's tool invocation system | High |
| **True multi-turn sessions** | Replay conversation turns as separate session messages instead of concatenating them | Medium |
| **Token usage** | Extract accurate prompt / completion token counts from session events | Medium |
| **Error mapping** | Map Copilot-specific error codes to typed `LanguageModelV3Error` subtypes | Low |
| **Unit tests** | Increase coverage of `convertToCopilotPrompt` edge cases | Low |
| **CI pipeline** | Add a GitHub Actions workflow for lint + build + test on PRs | Low |

If you are working on one of these, please open an issue first so we can
coordinate and avoid duplicating effort.
